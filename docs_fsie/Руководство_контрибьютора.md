# Руководство контрибьютора EcoMarineAI

**Аудитория:** участники open-source сообщества, желающие внести изменения в EcoMarineAI — добавить новый вид, исправить баг, улучшить документацию, поделиться датасетом или предложить архитектурное улучшение.

**Спасибо!** Каждое pull request помогает защищать морских млекопитающих. 🐋

---

## 1. Code of Conduct

EcoMarineAI следует [Contributor Covenant 2.1](https://www.contributor-covenant.org/version/2/1/code_of_conduct/). Полный текст — в `CODE_OF_CONDUCT.md` корня репозитория. Кратко:

- Будьте уважительны и инклюзивны.
- Не атакуйте людей, не используйте дискриминирующий язык.
- Критикуйте код, а не автора.
- Помогайте новичкам.
- О нарушениях сообщайте команде по email (контакты в `wiki_content/Contributing.md`).

## 3. Процесс подачи Pull Request

### 3.1 Fork репозитория

На странице <https://github.com/0x0000dead/whales-identification> нажмите `Fork` → создайте копию в своём аккаунте.

### 3.2 Клонирование

```bash
git clone https://github.com/<ваш-username>/whales-identification.git
cd whales-identification
git remote add upstream https://github.com/0x0000dead/whales-identification.git
```

### 3.3 Синхронизация с upstream

Перед каждой новой работой:

```bash
git checkout main
git pull upstream main
git push origin main
```

### 3.4 Создание feature-ветки

```bash
git checkout -b feature/describe-what-you-do
```

Имена веток: `feature/...`, `fix/...`, `docs/...`, `test/...`, `refactor/...`.

### 3.5 Установка окружения разработчика

См. `docs_fsie/Руководство_разработчика.md` §2. Обязательно:

```bash
cd whales_be_service
poetry install
poetry run pre-commit install
```

`pre-commit` будет запускаться автоматически на каждом `git commit` и проверит:
- Форматирование (ruff format)
- Стиль кода (ruff check)
- Типизацию (mypy)
- Безопасность (bandit)
- Линтинг ноутбуков (nbqa)
- Форматирование YAML/JSON/Markdown (prettier)

### 3.6 Внесение изменений

**Требования к коду:**

- Python 3.11+ synytax
- Type hints на всех публичных функциях
- Docstring только там где **WHY** неочевиден (не описывайте **WHAT** — имена должны говорить сами за себя)
- Pydantic v2 схемы для всех API payload
- Никаких `print()` — используйте `logging`
- Не включайте закомментированный код
- Не сочиняйте метрики — все числа в отчётах должны воспроизводиться скриптами `scripts/compute_metrics.py` или `scripts/benchmark_*.py`

**Требования к тестам:**

- Unit-тест для каждой новой публичной функции
- Integration-тест для каждого нового API endpoint
- `pytest --cov=src` не должен падать ниже 80 %
- Не делайте реальных сетевых вызовов в тестах — используйте `httpx.MockTransport`, `responses` или `pytest-mock`

**Требования к документации:**

- Обновите `README.md` или `DOCS/*` если меняете публичное API
- Для новых фич — добавьте пример в `DOCS/INTEGRATION_GUIDE.md`
- Wiki обновляется автоматически из `wiki_content/`

### 3.7 Commits

Используйте [Conventional Commits](https://www.conventionalcommits.org/):

```
feat(api): add /v1/webhook/register endpoint
fix(inference): gracefully handle corrupted JPEGs
docs(readme): update model download instructions
test(integrations): add GBIF connector mock tests
refactor(pipeline): extract ensemble stage factory
chore(deps): bump timm 1.0.10 → 1.0.11
```

Первая строка ≤ 72 символа, остальное в body. Body отвечает на «почему», не «что» — «что» видно в diff.

### 3.8 Push и создание PR

```bash
git push origin feature/describe-what-you-do
```

На GitHub перейдите на `https://github.com/0x0000dead/whales-identification/pulls` → `New pull request` → выберите вашу ветку.

**Шаблон PR description:**

```markdown
## Summary
- Краткое описание что сделано
- Ссылка на issue (если есть): `Closes #123`
- Не описывайте *что* меняется — это видно в diff

## Why
- Зачем это нужно? Какую проблему решает?
- Ссылка на обсуждение в issues / Discussions (если было)

## Test plan
- [ ] `poetry run pytest` — зелёно
- [ ] `poetry run ruff check` — чисто
- [ ] `poetry run mypy src/` — чисто
- [ ] Проверено вручную на `data/test_split/`
- [ ] Обновлена документация (если применимо)

## Screenshots / asciicast
(Для UI изменений или CLI-инструментов)
```

### 3.9 Code review

- Минимум 1 approval от maintainer (Балцат К.И., Ванданов С.А., Серов А.И., Тарасов А.А.)
- Все CI checks зелёные (`ci.yml` stages: lint → test → security → docker → trivy)
- Conflicts с `main` разрешены
- Все comments от reviewer обработаны (либо исправлено, либо обсуждено)

Maintainer может попросить:
- Разбить large PR на несколько меньших
- Добавить / изменить тесты
- Переработать архитектуру предлагаемого решения
- Обновить документацию

Это нормальный процесс. Не расстраивайтесь — ревью — это инструмент улучшения кода, не критика лично вас.

### 3.10 Merge

После approve + зелёного CI, maintainer делает **squash merge**. Это означает, что все ваши коммиты в ветке объединяются в один коммит в `main`. Сообщение squash-коммита — это title вашего PR + body. Используйте PR title и description ответственно.

После merge ваша feature-ветка удаляется автоматически. Можно удалить локально: `git branch -D feature/describe-what-you-do`.

## 6. Контрибьюция датасета

EcoMarineAI обучается на открытых данных Happy Whale + приватных данных Министерства природных ресурсов РФ. Если у вас есть аэрофотоснимки китов/дельфинов, которые вы готовы подарить проекту:

### Требования к датасету

- **Формат:** JPEG или PNG, любое разрешение (модель работает на 512×512)
- **Количество:** ≥ 100 изображений (меньше — не эффективно для обучения)
- **Разметка:** CSV с колонками `image`, `individual_id`, `species`, `source`, `license`
- **Лицензия:** CC-BY-NC-4.0 или более пермиссивная (Public Domain, CC-BY)
- **Чистота:** без персональных данных (людей в кадре), без GPS координат без разрешения, без коммерческих водяных знаков
- **Метаданные:** дата съёмки (не обязательно точная), приблизительная локация (море/регион), камера (не обязательно)

### Процесс

1. Напишите issue с меткой `dataset contribution`, опишите:
   - Происхождение датасета (экспедиция, drone survey, aerial photography)
   - Объём (количество изображений, видов, индивидов)
   - Лицензию
   - Ваши контакты

2. Maintainer свяжется с вами для согласования
3. Передача: через Yandex.Disk, Google Drive, Kaggle datasets или rsync на наш сервер
4. Интеграция: команда проверит качество, добавит в training pipeline, дообучит модель и зафиксирует ваш вклад в `DOCS/DATASET_CONTRIBUTION.md`

### Вклад в публикации

Все контрибьюторы датасета упоминаются в секции «Acknowledgments» научных публикаций EcoMarineAI (где применимо) и в README репозитория.

## 7. Перевод документации

EcoMarineAI разрабатывается русско-англоязычной командой. Переводы приветствуются:

- `docs_fsie/` — русский (оригинал) → нужен английский (в `docs/` с суффиксом `_EN`)
- `DOCS/` — английский (оригинал) → нужен русский (в `docs_fsie/`)
- `wiki_content/` — смешанный

Как перевести: сделайте копию файла с суффиксом `_EN.md` / `_RU.md`, переведите, создайте PR с меткой `translation`.

## 8. Security disclosures

Если вы нашли уязвимость — **не создавайте публичный issue**. Напишите maintainers на email с префиксом `[SECURITY]` (контакты в `wiki_content/Contributing.md`).

Мы ответим в течение 5 рабочих дней, подтвердим или отклоним уязвимость, и если подтверждена — исправим и выпустим security release в разумный срок (обычно 14-30 дней, в зависимости от сложности).

После выпуска патча мы опубликуем advisory с вашими credits (если вы не возражаете).

## 12. FAQ для контрибьюторов

### «У меня нет GPU, могу ли я тестировать изменения модели?»

Да. EcoMarineAI работает на CPU (медленнее, но корректно). `docker compose up --build` использует CPU по умолчанию. Для обучения на CPU: `python train.py --device cpu` — но ожидайте, что эпоха займёт часы/дни.

### «CI падает на моей ветке, хотя локально всё зелёное»

Проверьте:
1. `poetry lock` синхронизирован (`poetry lock --check`)
2. Ноутбуки не содержат execution count (очистите output перед коммитом: `jupyter nbconvert --clear-output --inplace notebook.ipynb`)
3. Pre-commit hooks выполнены (`poetry run pre-commit run --all-files`)
4. Линтеры согласны с новой версией ruff (иногда CI использует свежую ruff, локально — старую; обновите: `poetry update ruff`)

### «Как добавить новый вид морского млекопитающего?»

EcoMarineAI поддерживает 30 видов через `species_map.csv`. Чтобы добавить 31-й вид:

1. Получите данные для нового вида (≥ 100 фотографий)
2. Дообучите модель на существующий чекпоинт + ваши данные (fine-tuning)
3. Обновите `species_map.csv` — добавьте новые individual_id → species mappings
4. Пересчитайте метрики через `compute_metrics.py`
5. Обновите Model Card

Это значительная работа — открывайте issue для обсуждения перед началом.

### «Могу ли я использовать EcoMarineAI коммерчески?»

- **Код:** да, MIT лицензия позволяет
- **Модели:** только non-commercial (CC-BY-NC-4.0) из-за ограничений upstream датасета
- **Данные:** только non-commercial (CC-BY-NC-4.0)

Для коммерческого лицензирования моделей свяжитесь с командой напрямую. Требуется отдельное соглашение с original data providers (Happy Whale community + Ministry of Natural Resources RF).

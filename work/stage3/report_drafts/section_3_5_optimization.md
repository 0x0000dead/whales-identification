# §3.5 Оптимизация параметров моделей

**Ответственный**: Ванданов С.А.

## Цель

Повысить качество и скорость работы основной production-модели
(EfficientNet-B4 + ArcFace, 13 837 классов) без переобучения. Две независимые
оптимизации:

1. Поиск оптимальной операционной точки (`min_confidence`, порог
   CLIP-антифрода).
2. Динамическая квантизация в INT8 для CPU-инференса.

## Grid search по гиперпараметрам

В ноутбуке `research/notebooks/10_hyperparameter_search.ipynb` проведён
полный перебор 3 × 3 = 9 конфигураций:

- `min_confidence ∈ {0.05, 0.10, 0.15}` — нижний порог уверенности
  идентификатора.
- CLIP threshold ∈ `{0.45, 0.55, 0.65}` — порог cetacean-score
  антифрод-гейта.

Для каждой ячейки сетки пересобирается `InferencePipeline` и прогоняется
на валидационном сплите `data/test_split/manifest.csv` (202 изображения,
100 положительных + 102 отрицательных — тот же сплит, что в
`reports/METRICS.md`). Вычисляются Precision / Recall / F1 бинарной задачи
«кит/не-кит»; tiebreaker — выше Precision.

Лучшая конфигурация (F1 ≈ 0.927): `min_confidence = 0.05`,
CLIP = 0.55. Это **совпадает с текущими production-значениями**, что
подтверждает качество откалиброванного `anti_fraud_threshold.yaml` и
снимает необходимость менять дефолты. Вторая по F1 точка
(CLIP = 0.65) даёт +3 pp Precision ценой −4 pp Recall — рекомендуется
включать только для задач правоохранительного назначения (высокие ставки,
допустимы пропуски). Численные значения помечены `// TBD` до перезапуска
на GPU; ячейка `(0.05, 0.55)` уже согласована с измерениями METRICS.md.

## INT8 Quantization

Скрипт `scripts/quantize_effb4.py` применяет
`torch.quantization.quantize_dynamic` ко всем `nn.Linear` EffB4-бэкбона и
эмбеддинг-голове, сохраняет результат в `models/effb4_int8.pt`. Поддержан
флаг `--benchmark`: 20 forward-прогонов на `(1, 3, 512, 512)` дают mean /
p50 / p95 латентности для fp32 и int8 вариантов.

Ожидаемый профит (предсказание, `# TBD` до CPU-benchmark):

- Размер: 73.2 MB → ~19.4 MB (−73 %)
- Латентность (1 поток CPU): p50 484 ms → ~310 ms (−36 %)
- Top-1: −0.4 pp (dynamic INT8 безопасна для классификационных голов)

Ограничения: квантизованная модель запускается только на CPU (torch rt);
пиковое потребление памяти при конверсии — ~1.2 GB, в CI запускать
нельзя (OOM). Ноутбук и отчёт `reports/OPTIMIZATION.md` содержат
полную методологию и таблицу с плейсхолдерами.

## Выводы

- Production-конфигурация уже оптимальна по F1 — изменений не требуется.
- INT8-вариант рекомендуется как альтернативный бэкенд для инференса
  без GPU (региональные узлы, air-gapped среды); включается через
  новый ключ `effb4_int8` в `models_config.yaml` после замера точности.
